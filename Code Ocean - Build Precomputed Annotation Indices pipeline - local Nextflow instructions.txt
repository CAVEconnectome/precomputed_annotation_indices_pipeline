DOWNLOADING THE PIPELINE
----------

The pipeline can be downloaded to a local machine. It won't be run via Code Ocean locally, but rather simply via the underlying Nextflow.

First, install Nextflow, which itself will require installing Java:
  https://www.nextflow.io/docs/latest/install.html

Download the pipeline, which arrives a zip file, and unpack it. You will get a directory whose name includes the pipeline's name or tag whatever (the same number that appears in the pipeline's URL). For example, my pipeline looks like this:
pipeline-9094939/
  metadata/
    metadata.yml
  pipeline/
    main.nf
    nextflow.config
    nextflow-codeocean.config
  REPRODUCING.md

Read the REPRODUCING markdown file. It walks through running the pipeline pretty effectively. The main steps are:
  - Downloading data in the same format the pipeline was receiving it in Code Ocean.
  - Logging into the Docker registry.
  - Configuring the Code Ocean credentials.
  - Running the pipeline.

API KEYS, PYTHON ENVIRONMENT, DOCKER, SETTING UP THE ENVIRONMENT
----------

First, get an API key. In a browser, open Code Ocean and in the lower-left corner click your initials inside a circle and then click Account from the popup menu. The upper-left has three options: Credentials, User Secrets, and Access Tokens. Click Access Tokens. You should have at least one token already because you must have created one to use Code Ocean in the first place. However, there is no way to retrieve a token after you have created it. You can see them listed here, but you can't get their contents. If you saved your token when you created it, use that. Otherwise, go ahead and click 'Generate new token' in the upper-right. Give it a name and grant whatever accesses you want it to have. Then click Add Token. This will produce a new popup that makes the token available for you to copy. This is your only opportunity to do so. If you miss it, you'll have to trash the token and make another one (which is easy enough). Assuming you have copied the token, the next step will create a secret, which acts very similar. You give it a key (like a name) and copy the secret, just like you did with the token. Note that deleting tokens and secrets are done from the different options in the upper-left corner of the accounts view. Note that in my experience these two values (the token value and the secret value) are literally the exact same string. With the token and secret saved (and copied to some text file for subsequent use), return to the downloaded pipeline.

Open a terminal (shell) and navigate to the downloaded pipeline directory and then to the pipeline/ subdirectory within it. You must be inside that subdirectory to run the pipeline.

You're going to need some Python modules. Use pip:
  - pyarrow
This may not be necessary or appropriate. When I had this problem in a capsule, installing via pip in the main environment didn't alleviate the problem. I resorted to importing pyarrow within an exception block and refusing to support Parquet if pyarrow is unavailable. This is obviously not an ideal solution.

You're going to need Docker. I have found this depends on the local Mac Docker app (that you downloaded and installed at some earlier time, ahem). Run that app and log into DockerHub, or the Docker registry or whatever it is. If you leave the Docker app visible when you run the pipeline, you'll actually see the Docker GUI update on the fly, which is a good visual indication of at least partial success.

Back on the command line, following the steps in the REPRODUCING file to log into the Docker registry:
  $ docker login -u keith.wiley@alleninstitute.org registry.codeocean.allenneuraldynamics.org
This will prompt for a "password", but what it means is the access token you copied from the Code Ocean account settings. Copy/paste that in for the password.
If you get an error like this:
  request returned 500 Internal Server Error for API route and version . . .
then one way I have fixed it is to download an update for the Mac Docker app, install it, and restart the app (this is all-inclusive from the in-application's self-update tools under the Docker menubar option).

The next step indicated in the REPRODUCING document is to configure the Code Ocean credentials. It gives you the command:
  $ export CO_USERNAME=keith.wiley%40alleninstitute.org read -s -p "Enter Code Ocean password or API key: " CO_PASSWORD && echo && export CO_PASSWORD
For CO_PASSWORD it wants the same access token from above again. Go figure.
However, that command doesn't work on a Mac (for me at any rate) when running in the default zsh shell (as explained below). The "read" command fails, as shown here:
  $ export CO_USERNAME=keith.wiley%40alleninstitute.org read -s -p "Enter Code Ocean password or API key: " CO_PASSWORD && echo && export CO_PASSWORD
  export: not valid in this context: -s

That command is attempting to export CO_USERNAME, then prompt you for a password, then read your account password without showing it on the screen, and finally export the value as CO_PASSWORD. zsh shell doesn't support the -p behavior however, and even removing the prompt portion ('-p "Enter Code Ocean password or API key: "') doesn't work because export doesn't know what to do with the read command, so I did it using two separate commands:
  $ export CO_USERNAME=keith.wiley%40alleninstitute.org
  $ read -s CO_PASSWORD && export CO_PASSWORD
This worked for me.

CREDENTIALS
----------

From Code Ocean and Nextflow's perspective, you are now ready to run the pipeline. However, in ways that Code Ocean and Nextflow can't see, you actually aren't ready to run this particular pipeline yet. The annotation index pipeline relies on additional credentials that Code Ocean and Nextflow don't necessarily know about (they occur in the capsule code itself, which Code Ocean and Nextflow don't know to inspect before running the pipeline). Namely, some capsules upload data to Google buckets while other capsules download data from Google buckets. This is a technique I developed to avoid having Code Ocean pass data between consecutive capsules using its own built-in mechanism. Instead, some capsules in the pipeline, upload their results to GCP while the next capsules in the pipeline download those results from GCP, and data passed between the two capsules through Code Ocean is kept extremely minimal (or none at all). I designed the pipeline this way to achieve true MapReduce behaviors that Code Ocean can't facilitate (Nextflow might, but Code Ocean's simplified wrapping around Nextflow doesn't). Brief explanation: Code Ocean's "Collect" connection method passes all data from all upstream capsules to all downstream capsules, but MapReduce should conceptually divide the upstream data into labeled groups and only pass each group (a subset of the total data) to the downstream capsules that "subscribe" to a given label. My approach has all upstream capsules upload all their data to a GCP bucket, and then passes almost no data (or *absolutely* no data) to the downstream capsules. The downstream capsules then download only the data they need from the GCP buckets. All of this is to say that an exported local pipeline needs GCP credentials. Therefore you must export your google secret. Where does the secret come from? Most CAVE users, such as yourself, will have already stored them in:
  ~/.cloudvolume/secrets/google-secret.json
If you don't have that file, figure that out somehow. The next step is really tricky. You can't just export the contents of that file to the environment variable because Code Ocean will interpret/evaluate/convert all the syntax in the environment variable, most troublingly the quotes and such that the json secret contains. After much wrangling with escape characters and conversions and blah blah blah I gave up trying to solve this problem and instead base-64 encoded the whole secrets file to completely obscure its textual content when treating it as an environment variable. This means the capsule code *expects* the environment variable to have been base-64 encoded and will promptly base-64 decode it before using it. So we must base-64 encode the google secret file contents before pasting it into the GOOGLE_SECRET environment variable of the shell. I'll leave it to you to figure out how to do that, but it will end up with a command line command like the following:
  $ export GOOGLE_SECRET=<your_base64_encoded_google_secret>

The following is a Python script that will read and convert your cloudvolume google-secret.json to the required string. Run this, copy the result, and paste it to the command above:
import base64
with open("<PATH_TO_HOME_DIRECTORY>/.cloudvolume/secrets/google-secret.json") as f:
    google_secret_json = f.read()
google_secret_json_clean = google_secret_json.strip().replace('\n', ' ')
google_secret_json_clean_bytes = google_secret_json_clean.encode('utf-8')
google_secret_json_clean_bytes_b64encoded_bytes = base64.b64encode(google_secret_json_clean_bytes)
google_secret_b64encoded = google_secret_json_clean_bytes_b64encoded_bytes.decode('utf-8')
print(google_secret_b64encoded)

FIXING THE CONFIG FILES
----------

Code Ocean still hasn't prepared Nextflow correctly though! The google secrets environment variable still won't work yet! ARGH! Code Ocean bundles the exported pipeline with two config files, which you will find the pipeline/ directory:
  pipeline/nextflow.config
  pipeline/nextflow-codeocean.config
Open the Code Ocean specific configuration file. You'll see several lines that pass the GOOGLE_SECRET environment variable from the shell into the capsules. But they are missing critical dollar signs that would evaluate the literal string GOOGLE_SECRET as an environment variable! They look like this:
  containerOptions = '--env GOOGLE_SECRET=GOOGLE_SECRET'
When they should look like this instead:
  containerOptions = '--env GOOGLE_SECRET=$GOOGLE_SECRET'
Why oh why must we deal with this nonsense, amirite? Make that correction on all of the relevant lines.

SPECIFYING THE DATA SOURCE
----------

We need to specify the intended input data source. After all, if you exported the pipeline with a configuration that connected to a huge dataset, then that is what your local Nextflow will expect. You might not have even downloaded that particular dataset above where obtaining data was described. Open pipeline/main.nf. This describes the entire pipeline: all the capsules, all the data assets, and all the connections between them. Search for "Build precomputed annotation indices - decompress input". You are now looking at the configuration of the capsule that takes in the original input data. It is the only capsule that receives a direct connection from a specified input data source. This is where you need to make an alteration to process a different input data source. In the capsule configuration find the "ln" command. It's right below several "mkdir" commands. You need to alter the arguments to the "ln" command to point to your desired input data as downloaded and arranged in the data/ subdirectory, which you did above. For example, perhaps the "ln" command initially looks like this:
  ln -s "/tmp/data/synapses_pni_2_v1_filtered_view__v1412__test4" "capsule/data/synapses_pni_2_v1_filtered_view__v1412__test4" # id: fa661835-c8bc-4089-8eae-41ec1a2b4ea4
It is configured to process my "test4" dataset, a subset of my whole dataset. If I want to process a different dataset, I need to change this line to, perhaps:
  ln -s "/tmp/data/synapses_pni_2_v1_filtered_view__v1412__test1" "capsule/data/synapses_pni_2_v1_filtered_view__v1412__test1" # id: fa661835-c8bc-4089-8eae-41ec1a2b4ea4
Study the contents of your data/ subdirectory and make sure you indicate the intended data source on this line. Notice that the data source is reference twice in the "ln" command!

There is another step here. We must specify the data source as a command line argument to the pipeline. Open pipeine/nextflow-codeocean.config and find the params section:
  params {
      capsule_mapreduce_process_synapses_table_generate_config_28_args = "--data_source_name=unspecified"
  }
The "data_source_name" needs to be populated with the same name you just put in the capsule description. For our example above, you would replace "unspecified" with "synapses_pni_2_v1_filtered_view__v1412__test1".

RUNNING THE PIPELINE
----------

Whoa. We're done. You should be ready to run the pipeline now. To run the pipeline, the command is offered in REPRODUCING. However (wait, there's a however?), it is problematic because the command as offered in the REPRODUCING file sets the Nextflow version to 24.10.4, which won't work. So simply discard that from the beginning of the command, thusly:
  $ DATA_PATH=$PWD/../data nextflow -c nextflow-codeocean.config -log ../results/nextflow/nextflow.log run main.nf -resume

That *should* run your pipeline. If not, buena suerte en tu travesía amigo.

RUNNING THE PIPELINE AGAIN, PERHAPS WITH CODE ALTERATIONS
----------

The next problem is that the pipeline is not modifiable. What Code Ocean has done is download a milestoned version of your pipeline. If you edit the capsules, say in the Code Ocean GUI in the browser, and commit your changes, and rerun the locally downloaded pipeline, it won't pull your code changes! To make Nextflow pull updates to the capsules, open pipeline/main.nf. Peruse the file and see how it is laid out. It contains blocks for each capsule in the pipeline. Find the line in each block that pulls the capsule. It is a "git -C capsule-repo checkout" line. Each such line pulls a hash code (a random mix of letters and numbers):
  git -C capsule-repo checkout <SOME_LONG_HASH_STRING> --quiet
That hash code locks the downloaded pipeline to the milestoned state of the capsules at the time you downloaded the pipeline. To make the pipeline pull the latest version of every capsule, you must replace every instance of a hash in those lines with "HEAD" so the lines now look like this:
  git -C capsule-repo checkout HEAD --quiet
Making this change to upwards of a hundred entries in the file can be tedious. If you are adept at regular expression search/replace you can do it more easily, but it depends on your text editor of choice, so good luck. Perhaps a search string like this will work:
  git -C capsule-repo checkout [0-9a-z]+ --quiet
Change all instances to:
  git -C capsule-repo checkout HEAD --quiet
There is one more problem however. Running the pipeline will still attempt to use any currently downloaded capsules when you run it. It won't explicitly repull them. To force Nextflow to repull the capsules, remove the pipeline/work/ directory before rerunning the pipeline (it won't be there when you first download the pipeline, but will appear after you run the pipeline the first time). You must remove that work/ directory every time you run the pipeline if you want it to pull any changes to the capsules:
$ rm -rf work
Or you can combine the two commands into a one-liner (NOTE THAT THIS MAY FAIL!):
$ rm -rf work && DATA_PATH=$PWD/../data nextflow -log ../results/nextflow/nextflow.log run main.nf -resume
I find that the command above often fails ala "ln: /data: Read-only file system". If that occurs, try the original command from above, like this:
$ rm -rf work && DATA_PATH=$PWD/../data nextflow -c nextflow-codeocean.config -log ../results/nextflow/nextflow.log run main.nf -resume
